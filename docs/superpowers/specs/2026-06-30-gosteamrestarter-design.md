# GoSteamRestarter 设计说明

## 目标

GoSteamRestarter 是一个跨平台的 Steam 重启工具，目标是复刻现有批处理脚本的核心能力，并补齐 Windows、Linux、macOS 三端支持。第一版采用“终端核心 + 轻量桌面壳”的结构：业务逻辑全部复用，CLI 和 GUI 只是两种入口。

## 范围

### 第一版包含

- 结束 Steam 客户端进程
- 重启 Steam 客户端
- 清理系统 DNS 缓存
- 配置 Steam 启动路径
- 配置 Steam 启动参数
- 首次启动自动发现 Steam 安装位置
- 终端菜单界面
- 轻量桌面界面
- 跨平台构建支持：Windows、Linux、macOS

### 第一版不包含

- 托盘常驻
- 自动更新
- 多语言完整国际化框架
- 云同步配置
- 复杂的插件机制

## 总体结构

### `core`

`core` 是唯一业务中心，负责协调配置、平台调用和用户可见的操作结果。

职责包括：

- 加载和保存配置
- 首次启动时执行 Steam 路径发现
- 校验 Steam 路径是否存在
- 发起结束 Steam、启动 Steam、清理 DNS 的动作
- 将底层失败转换为结构化错误

### `platform`

`platform` 负责吸收系统差异，按操作系统拆分实现。

职责包括：

- 定位 Steam 默认安装路径
- 查找并结束 Steam 进程
- 启动 Steam 可执行文件
- 执行 DNS 清理
- 抽象管理员权限或权限不足的差异提示

建议采用 `windows`、`linux`、`darwin` 三个实现文件或子包，保持接口一致。

### `ui/cli`

终端界面负责复刻现有脚本的主要交互方式。

菜单项建议保留：

- 强制结束 Steam
- 强制重启 Steam
- 清理系统 DNS 缓存
- 选项
- 关于

CLI 只负责展示菜单、读取输入、渲染错误提示，不包含平台逻辑。

### `ui/desktop`

桌面壳是轻量封装层，提供按钮和基础设置入口。

职责包括：

- 触发同一套 `core` 动作
- 展示当前配置和状态
- 显示错误弹窗或提示条
- 编辑启动路径和启动参数

桌面壳不复制业务逻辑，也不直接实现系统调用。

## 配置设计

配置采用 JSON 文件，不再使用批处理脚本式配置。

建议字段：

- `steamPath`
- `steamArgs`

配置目录遵循各平台惯例：

- Windows：`%AppData%`
- Linux：`$XDG_CONFIG_HOME`，否则 `~/.config`
- macOS：`~/Library/Application Support`

首次启动时：

1. 读取配置
2. 如果已有有效 `steamPath`，直接进入主界面
3. 如果没有，尝试自动发现 Steam
4. 发现成功则提示用户确认
5. 发现失败则进入手动输入流程

## 平台行为

### Windows

- 可按注册表路径尝试自动发现 Steam
- 结束进程时以 `Steam.exe` 为目标
- DNS 清理使用系统命令
- 对管理员权限不足保持明确提示，不静默忽略

### Linux

- 默认查找 Steam 常见安装位置和桌面启动方式
- 进程名与路径检查需要兼容发行版差异
- DNS 清理使用对应发行版可用方案，优先给出明确失败原因

### macOS

- 优先查找 `/Applications/Steam.app`
- 进程结束和启动需兼容 `.app` 包
- DNS 清理使用系统命令并保留权限提示

## 运行流程

### 启动流程

1. 程序启动
2. 读取配置
3. 自动发现或校验 Steam 路径
4. 进入 CLI 或 GUI 主界面

### 结束 Steam

1. 检查 Steam 是否在运行
2. 发起结束进程动作
3. 将结果返回给界面层
4. 界面层展示成功或失败信息

### 重启 Steam

1. 调用结束 Steam
2. 调用启动 Steam
3. 使用已保存的启动参数
4. 返回统一结果

### 清理 DNS

1. 调用平台特定 DNS 清理实现
2. 返回成功、权限不足或失败原因
3. CLI 以文本提示，GUI 以弹窗提示

## 错误处理

所有失败都应转成稳定、可展示的错误类型，而不是让界面层直接接触底层系统错误。

建议至少覆盖：

- Steam 路径不存在
- Steam 进程未找到
- 启动失败
- DNS 清理失败
- 权限不足
- 配置读写失败

CLI 显示简洁文本，GUI 显示图形化提示，但错误来源保持一致。

## 测试策略

### 核心测试

- 配置读写
- 路径发现
- 命令拼装
- 错误映射

### 平台测试

- 各平台默认路径规则
- 进程名和启动方式规则
- DNS 清理命令返回值处理

### 界面测试

- CLI 菜单流转
- 基础输入校验
- GUI 的关键操作回调

### 构建验证

- Windows 构建
- Linux 构建
- macOS 构建

## 实施原则

- 业务逻辑集中在 `core`
- 平台差异只存在于 `platform`
- CLI 和 GUI 共享同一套动作接口
- v1 优先保证可用和一致，不提前引入复杂扩展

## 结论

这个设计保留了原批处理脚本的使用习惯，同时把跨平台差异、配置持久化和界面入口解耦。第一版应该先交付稳定的 CLI，再在同一核心上补一个轻量 GUI，避免两套逻辑分叉。
